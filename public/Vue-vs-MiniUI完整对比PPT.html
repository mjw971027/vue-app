📝 写法对比 ①：数据绑定 — getValue() vs v-model

MiniUI 方式：手动从 DOM 取值/赋值
// HTML — 每个控件靠 ID 定位
<input id="txtName" class="mini-textbox"/>
<select id="selDept" class="mini-combobox"/>

// JS — 手动取值（步骤多、易出错）
var txtName = mini.get("txtName");
var name = txtName.getValue();     // ① 找到控件 → ② 取值

var selDept = mini.get("selDept");
var deptId = selDept.getValue();

// ===== 手动赋值 =====
mini.get("txtName").setValue("张三");
mini.get("selDept").setValue("D001");

// ===== 提交表单 — 逐个字段取 =====
var form = new mini.Form("#form1");
var data = form.getData(true);

// ⚠️ 核心问题：
//   数据和 UI 是两份独立的
//   每次操作都要"找到控件 → 操作"
//   ID 拼错、控件名记错 → 运行时才发现
Vue 3 方式：v-model 双向绑定
<!-- 模板 — 声明式绑定，不需要 ID -->
<input v-model="form.name" />
<select v-model="form.deptId">
  <option v-for="d in depts" :value="d.id">
    {{ d.name }}
  </option>
</select>

// JS — 定义数据，UI 自动同步
const form = reactive({
  name: '',
  deptId: ''
});

// ===== 读数据：直接拿变量！ =====
console.log(form.name);    // 永远是最新值
console.log(form.deptId);

// ===== 写数据：直接赋值！ =====
form.name = '张三';          // UI 自动更新
form.deptId = 'D001';        // 下拉框自动选中

// ===== 提交：直接拿对象 =====
await api.submit({ data: {...form} });

// ⭐ 核心：数据和 UI 是同一份
//   不需要 mini.get()，不需要 getValue()
//   TypeScript 在编译时就能发现拼写错误

📝 写法对比 ②：条件渲染 & 列表渲染

MiniUI — 条件显隐：手动 display + innerHTML
// 根据登录状态，切换页面区域
var el = document.getElementById('msg');
if (isLogin) {
  el.style.display = 'block';
  el.innerHTML = '欢迎，' + userName;
} else {
  el.innerHTML = '<a href="/login">请登录</a>';
}
MiniUI — 列表：手动拼接 HTML 字符串
var html = '';
for (var i = 0; i < list.length; i++) {
  var item = list[i];
  html += '<tr>' +
    '<td>' + item.id + '</td>' +
    '<td>' + item.name + '</td>' +
    '</tr>';
}
$('#tbody').html(html);  // 全量替换
Vue 3 — 条件：v-if / v-else 声明式
<!-- 一行搞定，多状态也清晰 -->
<div v-if="isLogin">
  欢迎，{{ userName }}
</div>
<div v-else>
  <a href="/login">请登录</a>
</div>

<!-- v-show：频繁切换用（DOM还在） -->
<div v-show="visible">...</div>
Vue 3 — 列表：v-for 声明式 + 精确更新
<!-- 写一行模板，自动循环 -->
<tr v-for="item in list" :key="item.id">
  <td>{{ item.id }}</td>
  <td>{{ item.name }}</td>
</tr>

// 数据变了直接赋值，Vue 自动更新 DOM
// 只更新变化的那一行，不是全量替换

📝 写法对比 ③：更新数据后 — 手动刷 N 处 vs 自动同步

MiniUI：改完数据 → 手动更新每一处 UI
// 场景：审批通过了一个工装申请
order.maStatus = '03';

// ===== 需要手动更新所有相关 UI =====

// 1. 状态标签 → 改文字 + 改颜色
mini.get('statusBadge')
  .setValue('已完成');
document.getElementById('statusBadge')
  .className = 'badge badge-success';

// 2. 操作按钮 → 隐藏编辑/删除
mini.get('btnEdit').setVisible(false);
mini.get('btnDelete').setVisible(false);

// 3. 统计面板 → 重新加载
mini.get('completedCount')
  .setValue(recount());
grid.reload();

// ⚠️ 忘更新任何一处 → 数据不一致！
Vue 3：改完数据 → 一切自动更新
// 场景：审批通过了一个工装申请
order.value.maStatus = '03';

// ===== 什么都不用做！Vue 自动处理：=====

// 1. 状态标签 → computed 自动变色
const badgeColor = computed(() =>
  order.value.maStatus === '03'
    ? 'success' : 'warning'
);

// 2. 按钮 → v-if 自动隐藏
<button
  v-if="order.maStatus !== '03'"
  @click="handleEdit">编辑
</button>

// 3. 列表 → 数据变了自动重渲染
tableData.value = newData;

// ⭐ 所有地方自动同步，绝不会遗漏！

🧠 根本区别：声明式 vs 命令式编程

❌ 命令式（Imperative）：告诉电脑 怎么做
// ===== 任务：显示"欢迎，张三" =====
// 你必须一步步告诉电脑每个操作

var container = document
  .getElementById('app');       // 1.找容器
var div = document
  .createElement('div');         // 2.建元素
div.className = 'welcome';         // 3.设类名
div.textContent =
  '欢迎，' + userName;            // 4.设文本
container.appendChild(div);        // 5.挂上去

// ===== 更新 = 重复步骤 =====
div.textContent = '欢迎，李四';  // 再手动改
container.removeChild(div);      // 要删再手动
你描述的是过程：每一步计算机操作
✅ 声明式（Declarative）：告诉电脑 要什么
<!-- ===== 任务：显示"欢迎，张三" ===== -->
<!-- 直接描述你想要的最终结果 -->

<template>
  <div
    v-if="isLogin"
    class="welcome"
  >
    欢迎，{{ userName }}
  </div>
</template>

const isLogin = ref(true);
const userName = ref('张三');

// ===== 更新 = 改数据 =====
userName.value = '李四';  // 只改数据，UI 自动变
isLogin.value = false;   // UI 自动消失
你描述的是结果："这个位置显示什么"
生活类比： 命令式 = "去厨房→开冰箱→拿鸡蛋→磕开→倒碗里"（每一步怎么做）
声明式 = "我要一个煎蛋"（要什么结果）

⚡ Vue 3 核心机制：响应式系统（Reactivity）

ref() — 基础响应式
// 包装基本类型：string/number/boolean
const count = ref(0);
const loading = ref(false);

// 也能包装对象/数组
const user = ref<User | null>(null);
const list = ref<Item[]>([]);

// JS 中需要 .value
count.value++;                // 模板自动解包
user.value = { name: '李四' };
list.value = newData;         // 全部替换
适合：单个变量、API 返回数据、列表
reactive() — 对象响应式
// 包装对象，不需要 .value
const form = reactive({
  username: '', password: ''
});
const pagination = reactive({
  currentPage: 1, pageSize: 50
});

// 直接读写属性
form.username = 'admin';
pagination.currentPage = 2;

// ⚠️ 不可解构，会丢失响应式！
// const { username } = form; ← 错
适合：表单数据、配置对象、分页
computed() — 计算属性
// 从现有数据派生，有缓存！
const fullName = computed(() =>
  firstName.value + ' ' + lastName.value
);

// 项目实例：只读判断
const isReadOnly = computed(() =>
  !('00' === maStatus || '01' === maStatus)
);

// 缓存：依赖不变 → 不重算
// 访问 3 次 fullName → 只算 1 次
适合：需要缓存的计算、模板多次引用

📌 ref 详解 — Vue 3 最基础的响应式 API

ref 的本质
// ref 把普通值包装成一个响应式对象
const count = ref(0)
// count 不是数字，是一个 Ref 对象：{ value: 0 }
// 但 Vue 会自动追踪谁读了 count.value
// 当 count.value 变了，自动通知所有读者

const name = ref('张三')
const loading = ref(false)
const user = ref<User | null>(null)
const list = ref<Item[]>([])

// ===== JS 中必须用 .value =====
count.value = 10
name.value = '李四'
loading.value = true
list.value = newData   // 整个数组替换

// ===== 模板中自动解包！不用 .value =====
// <div>{{ count }}</div>   ← 显示 10
// <input v-model="name">  ← 双向绑定
⚠️ 常见误区
· 忘记写 .value：JS 里 count = 10 不会生效
· 模板里写了 .value：{{ count.value }} 没必要
· ref 包对象时：改属性需要 user.value.name = 'x'
你项目中最常见的 ref 用法
// ① 基础状态标识
const loading = ref(false)
const errorMsg = ref('')
const submitting = ref(false)

// ② API 返回的数据（可整体替换为 null）
const user = ref<UserInfo | null>(null)
const tableData = ref<ComponentItem[]>([])
const settings = ref<Setting[]>([])

// ③ 组件实例引用（模板 ref="xxx"）
const formRef = ref<FormInstance>()
const uploadRef = ref<{ submit: () => void }>()
formRef.value?.validate()
uploadRef.value?.submit()

// ④ Pinia store 中的状态
const token = ref(getToken())
const isLoggedIn = ref(isAuthenticated())
💡 ref vs 普通变量
let count = 0 → 改了模板不知道
const count = ref(0) → 改了模板自动刷新
ref 的唯一价值：让 Vue 知道这个值变了。
💡 判断口诀 + 注意事项
✅ 用 ref：单个值、整体替换、可为 null
✅ 用 reactive：表单/配置/分页等多字段对象
❌ 不推荐 ref：表单数据对象（全 .value 麻烦）
不确定 → 用 ref，它最通用

📦 reactive 详解 — 对象级别的响应式

reactive 的本质
// reactive 深度代理一个对象的所有属性
const form = reactive({
  username: '',
  password: '',
  remember: false
})

// ===== 直接读写属性！不需要 .value =====
form.username = 'admin'       // ✅ 响应式
form.password = '123456'
form.remember = true
console.log(form.username)   // 'admin'

// ===== 深层响应式 =====
const order = reactive({
  id: 1,
  items: [{ name: '工装A', qty: 2 }]
})
order.items[0].qty = 5  // ✅ 嵌套也响应式！
⚠️ reactive 的三大限制
① 不能整体替换：form = newObj 丢失响应式
② 不能解构：const {name} = form 丢失响应式
③ 只能放对象/数组：reactive(5) 报错
你项目中的 reactive 用法
// ① 登录表单（Login.vue）
const loginForm = reactive({
  username: '',
  password: ''
})

// ② 表单验证规则（Login.vue）
const rules = reactive<FormRules>({
  username: [{ required: true, message: '...' }],
  password: [{ required: true, min: 6 }]
})

// ③ 分页配置（Page4.vue）
const pagination = reactive({
  currentPage: 1,
  pageSize: 50,
  total: 0
})
pagination.currentPage = 2   // 直接改

// ④ 搜索条件（Page4.vue）
const searchForm = reactive<SearchParams>({
  companyId: '', deptId: '',
  startDate: '', endDate: ''
})
💡 reactive vs ref 选择规则
一组关联的配置/表单字段 → reactive
单个值/需要整体替换 → ref
追求写法简洁（不写 .value） → reactive
💡 判断口诀 + 注意事项
✅ 用 reactive：表单、验证、分页、配置（写法干净）
❌ 不推荐 reactive：API 返回数据、单个基本值、频繁解构
⚠️ 限制：不能整体替换、不能解构、只能对象/数组

// ③ fullName → 命中缓存（不打印）

🧮 computed 使用场景 — 什么时候该用？什么时候不该用？

🛡️ Vue Router — 路由守卫：前端三级权限控制

MiniUI 旧版：怎么控制权限？
🔴 登录检查：每个页面开头写一段 JS 检查 Session
🔴 权限检查：每个页面对照着权限表手动判断
🔴 菜单显示：后端渲染模板时决定显隐
🔴 容易遗漏：忘记在某个页面加检查 = 安全漏洞
Vue Router：全局守卫，一处检查
// beforeEach：所有路由跳转前统一拦截
router.beforeEach((to, from, next) => {

  // ① 已登录 → 访问登录页 → 直接跳首页
  if (to.path === '/login' && isAuth()) {
    next('/'); return;
  }

  // ② 未登录 → 访问需认证页 → 跳登录
  if (to.meta.requiresAuth && !isAuth()) {
    next({ path: '/login',
              query: { redirect: to.fullPath }});
    return;
  }

  // ③ 非管理员 → 访问管理员页 → 跳首页
  if (to.meta.requiresAdmin && !isAdmin()) {
    next('/'); return;
  }

  // ④ 无页面权限 → 跳首页
  const perm = to.meta.pagePermission;
  if (perm && !hasPerm(perm)) {
    next('/'); return;
  }

  next(); // ✅ 放行
});

// afterEach：自动设置页面标题
router.afterEach(to => {
  document.title = to.meta.title; });
⭐ 登录/角色/页面权限 三级连查，一处不漏。新增页面只需配置 meta，守卫自动生效。

🔄 Token 刷新队列 — 并发请求不丢失

问题场景
// 用户打开一个页面，同时加载 3 个接口：
// GET /api/components/list      ← 接口1
// GET /api/components/options   ← 接口2
// GET /api/user/me              ← 接口3

// 如果 Token 刚过期，3 个请求同时返回 401
// 每个都去调刷新接口 → 浪费！且可能冲突

// ===== MiniUI 时代的处理 =====
// 每个 $.ajax 的 error 回调各自处理
// 可能出现：A 刷新了 Token，B 和 C 不知道
// 继续用旧 Token 重试 → 仍然 401
Vue 3 的解决方案：请求队列
// 用一个 isRefreshing 标记 + 队列解决
let isRefreshing = false;
let queue: ((token: string) => void)[] = [];

if (status === 401) {
  if (isRefreshing) {
    // 已有刷新在进行中 → 排队等待
    return new Promise(resolve => {
      queue.push(token =>
        resolve(service(request)));
    });
  }
  isRefreshing = true;
  const newToken = await refreshToken();
  isRefreshing = false;
  // 通知所有排队的请求用新 Token 重试
  queue.forEach(fn => fn(newToken));
  queue = [];
  return service(originalRequest);
}
这在不依赖后端 Session 的 JWT 无状态认证体系里是标准做法。MiniUI 时代用 Session 没有这个问题，但 Session 不适用于前后端分离。

💀 传统一体式开发的五大痛点 — 每一个都是"慢性毒药"

为什么前后端一体是过去式、分离是必然？先看清旧模式到底痛在哪里。
🕐 痛点一：开发互锁，效率低下
场景前端改个按钮颜色 → 后端也要重新编译打包 → 前后端互相阻塞
后果前端等后端写好接口才能联调；后端等前端提需求才能改页面；双方相互等待
数据据统计，一体式项目中 40% 的开发时间花在"等待对方完成"上
🔒 痛点二：部署绑定，"一荣俱荣，一损俱损"
场景前端 CSS 里改了一行 → 整个 war 包重打 → 后端也一起重新部署
风险前端小改动可能引入后端回归测试；后端出 Bug 时前端服务也被迫停机
结果无法独立发版 → 发版流程沉重 → 迭代速度越来越慢
👥 痛点三：角色混乱，Java 开发被迫写前端
现状Java 开发从 Service 写到 JSP/HTML，一个人包揽全栈
代价后端逻辑质量下降（精力分散）；前端代码质量差（非专业前端）
恶性循环前端写得烂 → 后续改不动 → 越改越烂 → 没人敢维护
📉 痛点四：代码腐化，维护成本指数增长
表现同一段表单逻辑 copy 到 5 个页面；改一个 BUG 要改 5 处
原因HTML 没有组件化 → 复制粘贴是唯一的"复用"手段 → 代码量爆炸
后果维护难度 = 页面数 × 功能数；新增一个字段要改 N 个 HTML 文件
🧱 痛点五：无法多端扩展 — "锁死"了系统的未来
需求领导说"我们也要做个小程序版本"或"给领导做个大屏看板"
困境一体式架构下，后端逻辑和前端 HTML 混在一起，无法单独暴露 API → 小程序/看板没接口可用
结果要么重新写一套接口（重复劳动），要么放弃多端（业务受限）
🔴 核心结论： 前后端一体就像连体婴儿 — 做什么都要一起动，一个出问题另一个也遭殃。 这种模式在 2010 年代是主流，但在 2026 年的今天，已经是严重拖累团队效率的系统性架构债。

⏳ 技术债务的"利息效应" — 为什么不分离会越来越贵

技术债像信用卡：今天不还，明天的利息会吃掉你更多的预算。
📊 维护成本随页面数指数增长
# 一体式：每新增 1 页 → 独立 HTML + JS + CSS
  项目初期（5 页） → 维护成本 100%
  项目中期（20 页）→ 维护成本 400%+
  项目后期（50 页）→ 维护成本 1000%+
  # 关键：代码没有复用，每页都是独立"孤岛"

  项目初期（5 页） → 核心组件 3-5 个
  项目中期（20 页）→ 核心组件 8-12 个
  项目后期（50 页）→ 核心组件 15-20 个
  # 大部分页面是已有组件的排列组合
一体式复用率 ≈ 0%（每页复制粘贴）

❌ 安全风险Session 认证跨域受限，不适用于微服务
❌ 无法多端适配纯 HTML 页面，无法对接移动端/小程序
❌ 无构建工具代码质量无法保证，Bug 发现周期长

✅ 用户体验升级SPA 无刷新切换，HMR 热更新，TypeScript 安全
✅ JWT 无状态认证天然支持微服务架构和分布式部署
✅ 多端复用接口一套后端 API，PC/移动/小程序通用
✅ 提升系统寿命现代工具链保证 5-8 年可持续演进
